# /1c-feature-dev

Нативный workflow-скилл для полного цикла разработки фичи под стек `onec-mcp-universal`.

## Принцип

- Единственная точка одобрения: фаза дизайна.
- После утверждения дизайн-артефактов выполнение идёт автономно без лишних уточнений.
- Используются существующие onec tools/skills, без дублирования низкоуровневой логики.

## Режимы выполнения

- `express`: 1-2 объекта, низкий риск, без `tasks.md`
- `standard`: 3-7 изменений, `design.md + tasks.md`, последовательное выполнение
- `full`: 7+ изменений или архитектурная задача, `design.md + tasks.md`, декомпозиция на параллельные подпотоки

## 9 фаз

1. **Понимание проблемы**
   - Уточни цель и ожидаемый результат.
   - Собери контекст через `get_metadata`, `bsl_search_tool`, `graph_search/graph_related`.
2. **Scope и ограничения**
   - Границы изменений, совместимость, ограничения БСП/конфигурации.
3. **Capability mapping**
   - Привяжи каждую часть задачи к существующим tools/skills.
4. **Модель данных и метаданные**
   - Какие объекты 1С/реквизиты/регистры затрагиваются.
5. **Интеграция и миграции**
   - Влияние на формы, отчёты, внешние обработки, обратную совместимость.
6. **Стратегия проверки и отката**
   - Какие проверки обязательны и что делаем при провале.
7. **План задач (атомарно)**
   - Создай `tasks.md` (кроме `express`) с зависимостями, проверками и критериями Done.
8. **Исполнение**
   - Выполни шаги из плана в выбранном режиме.
9. **Отчёт о завершении**
   - Что сделано, что изменено, как проверено, остаточные риски.

## Обязательные проверки

- `validate_query` для всех новых/изменённых запросов
- `execute_code` для критичной бизнес-логики
- `find_references_to_object`/поиск по BSL для контроля сломанных ссылок
- профильные validate-скилы: `epf-validate`, `cf-validate`, `form-validate`, `meta-validate`, `skd-validate`, `role-validate` и т.д.

## Артефакты

- `design.md` (подход, объекты, риски, критерии)
- `tasks.md` (для `standard`/`full`)
- финальный отчёт (статус задач + список изменённых файлов)

## Правила

1. Не начинать фазу исполнения до утверждения `design.md`.
2. В каждой задаче должен быть проверяемый критерий готовности.
3. Не смешивать несвязанные изменения в одной задаче.
4. При падении проверок — исправление в той же ветке задач до зелёного статуса.
5. Все выводы фиксировать в артефактах, чтобы сессия была воспроизводимой через `/session-save`.
