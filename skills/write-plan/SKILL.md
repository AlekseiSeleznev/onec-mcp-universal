# /write-plan — План реализации

Создаёт `openspec/changes/<feature>/tasks.md` и превращает проектный дизайн в
исполняемый чеклист для дальнейшего выполнения.

## Вход

Аргумент:
- путь к `design.md`, например: `openspec/changes/my-feature/design.md`, или
- название фичи/изменения: `my-feature`.

Если аргумент не передан:
- ищи единственный каталог в `openspec/changes/` и используй его.

## Шаги

1. Определи `feature-id`:
   - `design.md` должен существовать.
2. Считай дизайн и требования качества из него.
3. Декомпозируй работу на задачи 2–5 минут (атомарные, без смешивания двух разных
   изменений в одной задаче).
4. Укажи зависимости между задачами (простыми ссылками `Задача 1 → Задача 2`).
5. Для каждой задачи добавь:
   - `Что делаем` (1–3 предложения);
   - `Файлы` и ожидаемые объекты;
   - `Скилл` (например, `/meta-compile`, `/cf-edit`, `/form-edit`);
   - верификационные критерии;
   - чекбокс `[ ]`.
6. В конце добавь задачу `Финальная валидация`:
   - `/cf-validate` или `/meta-validate`/`/form-validate`/`/mxl-validate` по типу затронутого артефакта;
   - проверка критериев из `design.md`.

## Формат `tasks.md`

Содержание должно быть читаемым и пригодным для следующего шага:

```markdown
# <Название фичи> — План реализации

**Цель:** <одно предложение>
**Дизайн:** openspec/changes/<feature>/design.md

## Задача N: <название>
**Статус:** [ ]
**Зависимости:** <нет/Задача X>
**Что делаем:** ...
**Файлы:** ...
**Скилл:** /meta-compile
**Проверка:** ...
```

## Результат

- Сохрани `openspec/changes/<feature>/tasks.md`;
- Сообщи пользователю о количестве задач и порядке выполнения;
- Предложи 2 режима:
  1. /openspec-apply <feature>
  2. Поэтапное ручное выполнение.

## Правила

1. Не включай задачи с неясными путями и без критерия Done.
2. Не менять логику/функциональность за пределами design-файла без явного подтверждения.
3. Для сложных задач не больше 10–12 задач; больше — делай разделение по подфичам.
