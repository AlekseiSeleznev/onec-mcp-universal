# /openspec-proposal — Формальное предложение изменения

Создай инициационный комплект для изменения, который можно потом передать в `/write-plan` или
`/openspec-apply` после согласования.

## Принцип

- Никакого кода до согласования proposal.
- Все требования должны быть трассируемы: что меняется и почему.
- Все файлы — в `openspec/changes/<feature>/`.

## Обязательные артефакты

- `openspec/changes/<feature>/proposal.md` — что меняем и зачем.
- `openspec/changes/<feature>/design.md` — подход и ограничения.
- `openspec/changes/<feature>/tasks.md` — черновой план (последующая нормализация через `/write-plan`).

## Процесс

1. Определи уникальный `feature-id` (латиницей/кириллическими транслитом, без пробелов).
2. Собери контекст:
   - `openspec/project.md`, если существует;
   - `openspec/changes/` и `openspec/specs/`, если есть;
   - связанные файлы дизайна через `get_metadata`, `bsl_search_tool`, `search`.
3. Создай каркас каталога `openspec/changes/<feature-id>/`.
4. Заполни `proposal.md`:
   - проблема;
   - цели;
   - границы изменений;
   - риски и критерии принятия.
5. Заполни `design.md`:
   - архитектурный подход;
   - affected objects (объекты/модули/формы);
   - требования к совместимости.
6. Создай `tasks.md` как предварительный список с пометками `[ ]`, но не выполняй.
7. Если есть формальные требования/спеки в проекте, укажи предполагаемые deltas в `proposal.md`.

## Форматирование (минимум)

`proposal.md`

- `## Goal`
- `## Scope`
- `## Risks`
- `## Acceptance`

`design.md`

- `## Current State`
- `## Target State`
- `## Data/Metadata Impact`

## Выход

- Сообщи пользователю итоговый путь `openspec/changes/<feature-id>/...`.
- Если пользователь согласовал, следующим шагом предложи `/write-plan <feature-id>`.

## Запреты

- Не начинай выполнять изменения до подтверждения `proposal.md`/`design.md`.
- Не использовать для быстрой идеи без структуры — для этого `/brainstorm`.
