# Паттерны запросов 1С

## ВАЖНО: Когда применять этот навык

Применяй ВСЕГДА когда:
- Пишешь запросы к базе данных (объект `Запрос`)
- Работаешь с виртуальными таблицами регистров (Остатки, Обороты, СрезПоследних)
- Делаешь выборку данных для обработки
- Формируешь отчёты или списки
- Обрабатываешь массивы данных

## Критический принцип

Каждый запрос = сетевой round-trip к СУБД.
Запрос в цикле из 1000 итераций = 1000 обращений к БД = катастрофа производительности.
ОДИН запрос вместо N запросов - главное правило оптимизации.

## ШАБЛОНЫ КОДА (копируй и адаптируй)

### Шаблон 1: Один запрос вместо цикла
```bsl
// Получаем ВСЕ данные одним запросом
Запрос = Новый Запрос;
Запрос.Текст =
"ВЫБРАТЬ
|   Номенклатура.Ссылка КАК Ссылка,
|   Номенклатура.Наименование КАК Наименование,
|   Номенклатура.Цена КАК Цена
|ИЗ
|   Справочник.Номенклатура КАК Номенклатура
|ГДЕ
|   Номенклатура.Ссылка В (&МассивСсылок)";

Запрос.УстановитьПараметр("МассивСсылок", МассивНоменклатуры);
Результат = Запрос.Выполнить();

// Создаём соответствие для быстрого доступа
СоответствиеДанных = Новый Соответствие;
Выборка = Результат.Выбрать();
Пока Выборка.Следующий() Цикл
    СоответствиеДанных.Вставить(Выборка.Ссылка, 
        Новый Структура("Наименование, Цена", Выборка.Наименование, Выборка.Цена));
КонецЦикла;

// Используем в цикле БЕЗ запросов
Для Каждого Строка Из Документ.Товары Цикл
    Данные = СоответствиеДанных.Получить(Строка.Номенклатура);
    Если Данные <> Неопределено Тогда
        Строка.Цена = Данные.Цена;
    КонецЕсли;
КонецЦикла;
```

### Шаблон 4: Пакетная обработка через ТаблицуЗначений
```bsl
// Подготавливаем таблицу значений

ТаблицаДанных.Колонки.Добавить("Штрихкод", Новый ОписаниеТипов("Строка", , Новый КвалификаторыСтроки(13)));

    НоваяСтрока = ТаблицаДанных.Добавить();
    НоваяСтрока.Штрихкод = Строка.Штрихкод;
    НоваяСтрока.Количество = Строка.Количество;
КонецЦикла;

// Один запрос обрабатывает все данные
Запрос = Новый Запрос;
Запрос.УстановитьПараметр("ДанныеИмпорта", ТаблицаДанных);
Запрос.Текст =
"ВЫБРАТЬ
|   Данные.Штрихкод,
|   Данные.Количество
|ПОМЕСТИТЬ втДанные
|ИЗ
|   &ДанныеИмпорта КАК Данные
|;
|
|////////////////////////////////////////////////////////////////////////////////
|ВЫБРАТЬ
|   Штрихкоды.Номенклатура,
|   втДанные.Количество
|ИЗ
|   втДанные
|       ВНУТРЕННЕЕ СОЕДИНЕНИЕ РегистрСведений.Штрихкоды КАК Штрихкоды
|       ПО втДанные.Штрихкод = Штрихкоды.Штрихкод";

Результат = Запрос.Выполнить();
```

## Таблица правил

| # | Правило | Обоснование |
|---|---------|-------------|
| 1 | Нет запросам в циклах | N round-trips к СУБД |
| 2 | Временные таблицы для сложных запросов | Читаемость, отладка, переиспользование |
| 3 | Параметризация запросов | Безопасность, кэширование планов |
| 4 | ЕСТЬNULL() при LEFT JOIN | NULL разрушает арифметику и условия |
| 5 | Параметры виртуальных таблиц | Оптимальный план запроса |
| 6 | Пакетные операции через ТаблицуЗначений | Один запрос вместо N |
| 7 | Выборка vs Выгрузка — выбирайте осознанно | Память vs удобство |
| 8 | Индексация полей соединения и фильтрации | Скорость поиска |
| 9 | Только нужные поля | Трафик, покрывающие индексы |
| 10 | ПЕРВЫЕ N для ограничения результата | Защита от OOM |
| 11 | Не маскируйте дубликаты через РАЗЛИЧНЫЕ | Лечите причину, не симптом |
| 12 | ВЫРАЗИТЬ для составных типов | Устранение лишних JOIN |
| 13 | ON vs WHERE в LEFT JOIN | LEFT vs фактический INNER |

## Правило 1: НИКОГДА не выполняйте запросы в цикле

**Почему:** Каждое выполнение запроса — это:
1. Сериализация параметров
2. Отправка по сети на сервер СУБД
3. Компиляция плана запроса (если не кэширован)
4. Выполнение
5. Возврат результата

Для N итераций цикла это N × (латентность сети + время выполнения). При 1000 элементов и латентности 5 мс — это **5 секунд** только на ожидание сети, даже если каждый запрос выполняется мгновенно.

**Стандарт ИТС:** «Запросы к базе данных — ограничение на использование запросов в цикле».

### Правильно — один запрос вместо N

```bsl
// Получаем все данные одним запросом
Запрос = Новый Запрос;
Запрос.Текст =
"ВЫБРАТЬ
|   Номенклатура.Ссылка КАК Ссылка,
|   Номенклатура.Наименование КАК Наименование,
|   Номенклатура.ЕдиницаИзмерения КАК ЕдиницаИзмерения,
|   Номенклатура.СтавкаНДС КАК СтавкаНДС
|ИЗ
|   Справочник.Номенклатура КАК Номенклатура
|ГДЕ
|   Номенклатура.Ссылка В (&МассивНоменклатуры)";

Запрос.УстановитьПараметр("МассивНоменклатуры", МассивНоменклатуры);
Результат = Запрос.Выполнить();
Выборка = Результат.Выбрать();

// Обрабатываем результат в цикле — запрос уже выполнен
СоответствиеДанных = Новый Соответствие;
Пока Выборка.Следующий() Цикл
    СоответствиеДанных.Вставить(Выборка.Ссылка, 
        Новый Структура("Наименование, ЕдиницаИзмерения, СтавкаНДС",
            Выборка.Наименование, Выборка.ЕдиницаИзмерения, Выборка.СтавкаНДС));
КонецЦикла;

// Используем полученные данные
Для Каждого СтрокаТоваров Из Документ.Товары Цикл
    ДанныеНоменклатуры = СоответствиеДанных.Получить(СтрокаТоваров.Номенклатура);
    Если ДанныеНоменклатуры <> Неопределено Тогда
        СтрокаТоваров.ЕдиницаИзмерения = ДанныеНоменклатуры.ЕдиницаИзмерения;
    КонецЕсли;
КонецЦикла;
```

### Неправильно — запрос в цикле

```bsl
// ПЛОХО: N итераций = N запросов к СУБД
Для Каждого СтрокаТоваров Из Документ.Товары Цикл
    
    Запрос = Новый Запрос;
    Запрос.Текст =
    "ВЫБРАТЬ
    |   Номенклатура.ЕдиницаИзмерения
    |ИЗ
    |   Справочник.Номенклатура КАК Номенклатура
    |ГДЕ
    |   Номенклатура.Ссылка = &Номенклатура";
    
    Запрос.УстановитьПараметр("Номенклатура", СтрокаТоваров.Номенклатура);
    Результат = Запрос.Выполнить();
    // ... обработка ...
    
КонецЦикла; // Для 1000 строк — 1000 запросов!
```

## Правило 2: Используйте временные таблицы для сложных запросов

**Почему:** Сложные запросы с многоуровневыми вложенными подзапросами:
1. Трудно читать и отлаживать
2. Оптимизатор СУБД может выбрать неоптимальный план выполнения
3. Невозможно переиспользовать промежуточный результат

Временные таблицы решают все три проблемы: запрос разбивается на этапы, каждый этап можно проверить отдельно, промежуточные результаты индексируются.

### Правильно — пакетный запрос с временными таблицами

```bsl
Запрос = Новый Запрос;
Запрос.Текст =
"ВЫБРАТЬ
|   Реализация.Ссылка КАК ДокументСсылка,
|   Реализация.Контрагент КАК Контрагент,
|   Реализация.СуммаДокумента КАК Сумма
|ПОМЕСТИТЬ втРеализации
|ИЗ
|   Документ.РеализацияТоваровУслуг КАК Реализация
|ГДЕ
|   Реализация.Дата МЕЖДУ &ДатаНачала И &ДатаОкончания
|   И Реализация.Проведен
|;
|
|////////////////////////////////////////////////////////////////////////////////
|ВЫБРАТЬ
|   втРеализации.Контрагент КАК Контрагент,
|   СУММА(втРеализации.Сумма) КАК ОбщаяСумма,
|   КОЛИЧЕСТВО(РАЗЛИЧНЫЕ втРеализации.ДокументСсылка) КАК КоличествоДокументов
|ПОМЕСТИТЬ втИтогиПоКонтрагентам
|ИЗ
|   втРеализации
|
|СГРУППИРОВАТЬ ПО
|   втРеализации.Контрагент
|;
|
|////////////////////////////////////////////////////////////////////////////////
|ВЫБРАТЬ
|   втИтоги.Контрагент КАК Контрагент,
|   втИтоги.ОбщаяСумма КАК ОбщаяСумма,
|   втИтоги.КоличествоДокументов КАК КоличествоДокументов,
|   ВзаиморасчетыОстатки.СуммаОстаток КАК Задолженность
|ИЗ
|   втИтогиПоКонтрагентам КАК втИтоги
|       ЛЕВОЕ СОЕДИНЕНИЕ РегистрНакопления.ВзаиморасчетыСКонтрагентами.Остатки КАК ВзаиморасчетыОстатки
|       ПО втИтоги.Контрагент = ВзаиморасчетыОстатки.Контрагент
|
|УПОРЯДОЧИТЬ ПО
|   ОбщаяСумма УБЫВ";

Запрос.УстановитьПараметр("ДатаНачала", ДатаНачала);
Запрос.УстановитьПараметр("ДатаОкончания", ДатаОкончания);

Результат = Запрос.Выполнить();
```

### Правила для временных таблиц

1. **Префикс `вт`** (временная таблица) — стандартное соглашение, повышает читаемость: `втОстаткиТоваров`, `втДокументыКОбработке`
2. **Индексируйте** временные таблицы, если соединяете их с другими: `ПОМЕСТИТЬ втДокументы ... ИНДЕКСИРОВАТЬ ПО Контрагент`
3. **Уничтожайте** временные таблицы, если менеджер временных таблиц не уничтожится автоматически: `МенеджерВТ = Новый МенеджерВременныхТаблиц`

```bsl
// Менеджер временных таблиц — управляет жизненным циклом
МенеджерВТ = Новый МенеджерВременныхТаблиц;
Запрос = Новый Запрос;
Запрос.МенеджерВременныхТаблиц = МенеджерВТ;

// ... выполнение пакета запросов ...

// Явное уничтожение или выход из области видимости переменной МенеджерВТ
```

## Правило 3: Параметризация запросов — не подставляйте значения в текст

**Почему:** Подстановка значений напрямую в текст запроса:
1. **Уязвимость** — аналог SQL injection (в контексте 1С менее критично, но всё равно опасно для строковых значений)
2. **Невозможность кэширования** — СУБД не может переиспользовать план запроса, так как текст каждый раз разный
3. **Ошибки форматирования** — даты, числа с разделителями, строки с кавычками — источники ошибок

### Правильно

```bsl
Запрос = Новый Запрос;
Запрос.Текст =
"ВЫБРАТЬ
|   Товары.Наименование
|ИЗ
|   Справочник.Номенклатура КАК Товары
|ГДЕ
|   Товары.ВидНоменклатуры = &ВидНоменклатуры
|   И Товары.Цена >= &МинимальнаяЦена";

Запрос.УстановитьПараметр("ВидНоменклатуры", Перечисления.ВидыНоменклатуры.Товар);
Запрос.УстановитьПараметр("МинимальнаяЦена", 1000);

Результат = Запрос.Выполнить();
```

### Неправильно

```bsl
// ПЛОХО: подстановка значений в текст запроса
Запрос = Новый Запрос;
Запрос.Текст =
"ВЫБРАТЬ
|   Товары.Наименование
|ИЗ
|   Справочник.Номенклатура КАК Товары
|ГДЕ
|   Товары.Наименование ПОДОБНО ""%" + СтрокаПоиска + "%"""; // Опасно! Что если СтрокаПоиска содержит кавычки?
```

## Правило 4: Обработка NULL в LEFT JOIN

**Почему:** `ЛЕВОЕ СОЕДИНЕНИЕ` возвращает NULL для полей правой таблицы, если нет соответствия. В 1С NULL — это особое значение: `NULL + 100 = NULL`, `NULL > 0 = ЛОЖЬ`. Необработанный NULL приводит к:
1. Неверным расчётам (сумма с NULL = NULL)
2. Потерянным строкам в условиях (NULL не проходит условие `> 0`)
3. Ошибкам при дальнейшей обработке в коде

### Правильно — ЕСТЬNULL()

```bsl
Запрос.Текст =
"ВЫБРАТЬ
|   Номенклатура.Ссылка КАК Номенклатура,
|   Номенклатура.Наименование КАК Наименование,
|   ЕСТЬNULL(ОстаткиТоваров.КоличествоОстаток, 0) КАК Остаток,
|   ЕСТЬNULL(ОстаткиТоваров.СуммаОстаток, 0) КАК СуммаОстатка
|ИЗ
|   Справочник.Номенклатура КАК Номенклатура
|       ЛЕВОЕ СОЕДИНЕНИЕ РегистрНакопления.ТоварыНаСкладах.Остатки КАК ОстаткиТоваров
|       ПО Номенклатура.Ссылка = ОстаткиТоваров.Номенклатура
|ГДЕ
|   ЕСТЬNULL(ОстаткиТоваров.КоличествоОстаток, 0) > 0";
```

### Неправильно

```bsl
// ПЛОХО: без ЕСТЬNULL при LEFT JOIN — числовые поля могут быть NULL
Запрос.Текст =
"ВЫБРАТЬ
|   Номенклатура.Наименование,
|   ОстаткиТоваров.КоличествоОстаток КАК Остаток  // Будет NULL, если нет остатков!
|ИЗ
|   Справочник.Номенклатура КАК Номенклатура
|       ЛЕВОЕ СОЕДИНЕНИЕ РегистрНакопления.ТоварыНаСкладах.Остатки КАК ОстаткиТоваров
|       ПО Номенклатура.Ссылка = ОстаткиТоваров.Номенклатура
|ГДЕ
|   ОстаткиТоваров.КоличествоОстаток > 0";  // NULL > 0 = ЛОЖЬ → строки без остатков пропадут!
```

## Правило 5: Виртуальные таблицы регистров — передавайте параметры внутрь

**Почему:** Виртуальные таблицы регистров (`Остатки`, `ОстаткиИОбороты`, `Обороты`, `СрезПоследних`) — это не реальные таблицы, а параметризованные функции СУБД. Если параметры (период, отбор) задать внутри виртуальной таблицы — СУБД использует оптимальный план. Если фильтровать через WHERE — СУБД сначала вычислит **все** данные, а потом отфильтрует. Разница может быть в тысячи раз.

**Стандарт ИТС:** «Использование виртуальных таблиц» — параметры отбора виртуальных таблиц задавать в параметрах, а не в условии ГДЕ.

### Правильно — параметры внутри виртуальной таблицы

```bsl
Запрос.Текст =
"ВЫБРАТЬ
|   Остатки.Номенклатура КАК Номенклатура,
|   Остатки.Склад КАК Склад,
|   Остатки.КоличествоОстаток КАК Остаток
|ИЗ
|   РегистрНакопления.ТоварыНаСкладах.Остатки(
|       &ДатаОстатков,
|       Номенклатура В (&СписокНоменклатуры)
|           И Склад = &Склад
|   ) КАК Остатки";

Запрос.УстановитьПараметр("ДатаОстатков", ДатаОстатков);
Запрос.УстановитьПараметр("СписокНоменклатуры", МассивНоменклатуры);
Запрос.УстановитьПараметр("Склад", Склад);
```

### Неправильно — фильтрация через WHERE

```bsl
// ПЛОХО: СУБД сначала вычислит остатки по ВСЕЙ номенклатуре на ВСЕХ складах,
// а потом отфильтрует. На большой базе — минуты вместо миллисекунд.
Запрос.Текст =
"ВЫБРАТЬ
|   Остатки.Номенклатура,
|   Остатки.Склад,
|   Остатки.КоличествоОстаток
|ИЗ
|   РегистрНакопления.ТоварыНаСкладах.Остатки КАК Остатки
|ГДЕ
|   Остатки.Номенклатура В (&СписокНоменклатуры)
|   И Остатки.Склад = &Склад";
```

## Правило 6: Пакетные операции — ТаблицаЗначений + запрос вместо поэлементной обработки

**Почему:** Когда нужно обработать массив данных (например, загрузка из внешней системы), передача данных во временную таблицу через `Запрос.УстановитьПараметр("ВТ", ТаблицаЗначений)` позволяет обработать все данные одним запросом вместо цикла.

### Правильно — передача ТаблицыЗначений как параметра

```bsl
// Подготавливаем таблицу значений с данными для обработки
ТаблицаДанных = Новый ТаблицаЗначений;
ТаблицаДанных.Колонки.Добавить("Штрихкод", Новый ОписаниеТипов("Строка", , Новый КвалификаторыСтроки(13)));
ТаблицаДанных.Колонки.Добавить("Количество", Новый ОписаниеТипов("Число", Новый КвалификаторыЧисла(15, 3)));

Для Каждого СтрокаИмпорта Из ДанныеИмпорта Цикл
    НоваяСтрока = ТаблицаДанных.Добавить();
    НоваяСтрока.Штрихкод = СтрокаИмпорта.Штрихкод;
    НоваяСтрока.Количество = СтрокаИмпорта.Количество;
КонецЦикла;

// Один запрос обрабатывает все данные через временную таблицу
Запрос = Новый Запрос;
Запрос.УстановитьПараметр("ДанныеИмпорта", ТаблицаДанных);

Запрос.Текст =
"ВЫБРАТЬ
|   Данные.Штрихкод КАК Штрихкод,
|   Данные.Количество КАК Количество
|ПОМЕСТИТЬ втДанныеИмпорта
|ИЗ
|   &ДанныеИмпорта КАК Данные
|;
|
|////////////////////////////////////////////////////////////////////////////////
|ВЫБРАТЬ
|   Штрихкоды.Номенклатура КАК Номенклатура,
|   втДанные.Количество КАК Количество
|ИЗ
|   втДанныеИмпорта КАК втДанные
|       ВНУТРЕННЕЕ СОЕДИНЕНИЕ РегистрСведений.ШтрихкодыНоменклатуры КАК Штрихкоды
|       ПО втДанные.Штрихкод = Штрихкоды.Штрихкод";

Результат = Запрос.Выполнить();
```

### Неправильно

```bsl
// ПЛОХО: поиск по штрихкоду в цикле — N запросов
Для Каждого СтрокаИмпорта Из ДанныеИмпорта Цикл
    
    Запрос = Новый Запрос;
    Запрос.Текст = "ВЫБРАТЬ Штрихкоды.Номенклатура ИЗ РегистрСведений.ШтрихкодыНоменклатуры КАК Штрихкоды ГДЕ Штрихкоды.Штрихкод = &Штрихкод";
    Запрос.УстановитьПараметр("Штрихкод", СтрокаИмпорта.Штрихкод);
    // ... выполнение для КАЖДОЙ строки ...
    
КонецЦикла;
```

## Правило 7: Обработка результатов запроса — Выборка vs Выгрузка

**Почему:** Результат запроса можно обработать двумя способами: через выборку (курсор) и через выгрузку в ТаблицуЗначений. Выбор способа влияет на потребление памяти и удобство работы.

### Выборка (Выбрать/Следующий) — для последовательной обработки

```bsl
// Выборка экономит память: данные загружаются порциями
Результат = Запрос.Выполнить();
Выборка = Результат.Выбрать();

Пока Выборка.Следующий() Цикл
    // Обработка строки
    Сообщение = СтрШаблон("Товар: %1, Остаток: %2",
        Выборка.Наименование, Выборка.Остаток);
КонецЦикла;
```

### Выборка с группировкой (ВыборкаДетальныеЗаписи) — для иерархических данных

```bsl
// Двухуровневая обработка: группировка по контрагенту, детали — документы
Результат = Запрос.Выполнить();
ВыборкаКонтрагенты = Результат.Выбрать(ОбходРезультатаЗапроса.ПоГруппировкам);

Пока ВыборкаКонтрагенты.Следующий() Цикл
    
    // Уровень группировки — контрагент
    ИтогоПоКонтрагенту = ВыборкаКонтрагенты.Сумма;
    
    ВыборкаДокументы = ВыборкаКонтрагенты.Выбрать();
    Пока ВыборкаДокументы.Следующий() Цикл
        // Детальные записи — документы данного контрагента
        ОбработатьДокумент(ВыборкаДокументы.ДокументСсылка, ВыборкаДокументы.Сумма);
    КонецЦикла;
    
КонецЦикла;
```

### Выгрузка в ТаблицуЗначений — для дальнейших манипуляций

```bsl
// Выгрузка загружает ВСЕ данные в память. Используйте когда:
// - нужен произвольный доступ к строкам (поиск, сортировка)
// - результат передаётся в другую процедуру
// - данных гарантированно немного (< 10 000 строк)
ТаблицаРезультат = Запрос.Выполнить().Выгрузить();

// Поиск по ТаблицеЗначений
НайденнаяСтрока = ТаблицаРезультат.Найти(ИскомаяНоменклатура, "Номенклатура");
```

## Правило 8: Индексация — помогайте оптимизатору

**Почему:** Индексы в 1С создаются автоматически для ключевых полей (Ссылка, измерения регистров, код, наименование). Но для произвольных запросов с условиями по нестандартным полям индекса может не быть, и СУБД выполнит полное сканирование таблицы (full table scan). Для таблиц с миллионами записей это катастрофа.

### Где нужны индексы

1. **Поля в условиях ГДЕ** — если часто фильтруете по полю, оно должно быть проиндексировано
2. **Поля в условиях соединения** — ОН (ON) использует индексы
3. **Поля упорядочивания** — УПОРЯДОЧИТЬ ПО может использовать индекс

### Временные таблицы — индексируйте поля соединения

```bsl
// Индекс на временной таблице — критически важно для JOIN
Запрос.Текст =
"ВЫБРАТЬ
|   ДанныеЗаказов.Номенклатура КАК Номенклатура,
|   ДанныеЗаказов.Количество КАК Количество
|ПОМЕСТИТЬ втЗаказы
|ИЗ
|   &ТаблицаЗаказов КАК ДанныеЗаказов
|
|ИНДЕКСИРОВАТЬ ПО
|   Номенклатура
|;
|
|////////////////////////////////////////////////////////////////////////////////
|ВЫБРАТЬ
|   втЗаказы.Номенклатура,
|   втЗаказы.Количество КАК Заказано,
|   ЕСТЬNULL(Остатки.КоличествоОстаток, 0) КАК НаСкладе
|ИЗ
|   втЗаказы
|       ЛЕВОЕ СОЕДИНЕНИЕ РегистрНакопления.ТоварыНаСкладах.Остатки(,
|           Номенклатура В (ВЫБРАТЬ втЗаказы.Номенклатура ИЗ втЗаказы)) КАК Остатки
|       ПО втЗаказы.Номенклатура = Остатки.Номенклатура";
```

### Настройка индексов в конфигурации

В конфигураторе/EDT: Реквизит → Свойства → «Индексировать» = «Индексировать» или «Индексировать с дополнительным упорядочиванием».

## Правило 9: Не используйте ВЫБРАТЬ * (выбирайте только нужные поля)

**Почему:** Запрос `ВЫБРАТЬ *` (в 1С — `ВЫБРАТЬ Таблица.*`) возвращает все поля таблицы. Проблемы:
1. **Избыточный трафик** — большие поля (ХранилищеЗначения, длинные строки) передаются, даже если не нужны
2. **Хрупкий код** — при добавлении реквизита в конфигурацию запрос начнёт возвращать больше данных, код обработки может сломаться
3. **Невозможность использования покрывающего индекса** — СУБД вынуждена читать полные строки, а не только индекс

### Правильно

```bsl
// Явно перечисляем только нужные поля
Запрос.Текст =
"ВЫБРАТЬ
|   Контрагенты.Ссылка,
|   Контрагенты.Наименование,
|   Контрагенты.ИНН
|ИЗ
|   Справочник.Контрагенты КАК Контрагенты
|ГДЕ
|   Контрагенты.ЭтоГруппа = ЛОЖЬ";
```

### Неправильно

```bsl
// ПЛОХО: возвращает ВСЕ реквизиты, включая логотип (ХранилищеЗначения), 
// контактную информацию и десятки других полей
Запрос.Текст =
"ВЫБРАТЬ
|   Контрагенты.*
|ИЗ
|   Справочник.Контрагенты КАК Контрагенты";
```

## Правило 10: Ограничение результата и пагинация

**Почему:** Запрос без ограничения может вернуть миллионы строк, исчерпав оперативную память сервера приложений. Для отображения пользователю обычно нужны первые 100-1000 записей.

### Правильно — ПЕРВЫЕ N

```bsl
// Ограничиваем количество возвращаемых строк
Запрос.Текст =
"ВЫБРАТЬ ПЕРВЫЕ 100
|   Товары.Наименование,
|   Товары.Код
|ИЗ
|   Справочник.Номенклатура КАК Товары
|ГДЕ
|   Товары.Наименование ПОДОБНО &СтрокаПоиска
|УПОРЯДОЧИТЬ ПО
|   Товары.Наименование";

Запрос.УстановитьПараметр("СтрокаПоиска", СтрокаПоиска + "%");
```

### Правильно — запрос для динамического списка

```bsl
// Динамические списки автоматически поддерживают пагинацию через платформу.
// Не делайте свою пагинацию для данных, которые отображаются в динамическом списке.
```

## Правило 11: Избегайте РАЗЛИЧНЫЕ без необходимости

**Почему:** `ВЫБРАТЬ РАЗЛИЧНЫЕ` (аналог `SELECT DISTINCT`) требует сортировки или хеширования всех строк результата для устранения дубликатов. Если дубликатов нет или они допустимы — `РАЗЛИЧНЫЕ` тратит ресурсы впустую. Если дубликаты появляются — часто это признак неправильного JOIN.

### Правильно — устраните причину дубликатов

```bsl
// Если дубликаты из-за лишнего JOIN — исправьте запрос, а не маскируйте РАЗЛИЧНЫЕ
Запрос.Текст =
"ВЫБРАТЬ
|   Контрагенты.Ссылка,
|   Контрагенты.Наименование
|ИЗ
|   Справочник.Контрагенты КАК Контрагенты
|ГДЕ
|   Контрагенты.Ссылка В
|       (ВЫБРАТЬ РАЗЛИЧНЫЕ Реализация.Контрагент
|        ИЗ Документ.РеализацияТоваровУслуг КАК Реализация
|        ГДЕ Реализация.Дата >= &ДатаНачала)";
```

### Неправильно

```bsl
// ПЛОХО: РАЗЛИЧНЫЕ маскирует проблему дубликатов из-за лишнего JOIN
Запрос.Текст =
"ВЫБРАТЬ РАЗЛИЧНЫЕ
|   Контрагенты.Наименование
|ИЗ
|   Справочник.Контрагенты КАК Контрагенты
|       СОЕДИНЕНИЕ Документ.РеализацияТоваровУслуг КАК Реализация
|       ПО Контрагенты.Ссылка = Реализация.Контрагент";
// Контрагент с 10 реализациями — 10 строк, потом РАЗЛИЧНЫЕ убирает 9. Лишняя работа.
```

## Правило 12: Конструкция ВЫРАЗИТЬ — для оптимизации составных типов

**Почему:** Если поле имеет составной тип (например, «Регистратор» регистра может быть любым документом), СУБД выполняет LEFT JOIN ко ВСЕМ таблицам составного типа. `ВЫРАЗИТЬ(Поле КАК Тип)` ограничивает соединения только указанным типом, радикально улучшая производительность.

**Стандарт ИТС:** «Использование конструкции ВЫРАЗИТЬ в запросах».

### Правильно

```bsl
// ВЫРАЗИТЬ — ограничиваем тип, СУБД делает JOIN только с одной таблицей
Запрос.Текст =
"ВЫБРАТЬ
|   Движения.Период,
|   ВЫРАЗИТЬ(Движения.Регистратор КАК Документ.РеализацияТоваровУслуг).Контрагент КАК Контрагент,
|   Движения.Количество
|ИЗ
|   РегистрНакопления.ТоварыНаСкладах КАК Движения
|ГДЕ
|   Движения.Регистратор ССЫЛКА Документ.РеализацияТоваровУслуг";
```

### Неправильно

```bsl
// ПЛОХО: обращение к реквизиту через составной тип без ВЫРАЗИТЬ
// СУБД сделает LEFT JOIN к КАЖДОМУ типу, входящему в «Регистратор»
Запрос.Текст =
"ВЫБРАТЬ
|   Движения.Период,
|   Движения.Регистратор.Контрагент КАК Контрагент,
|   Движения.Количество
|ИЗ
|   РегистрНакопления.ТоварыНаСкладах КАК Движения";
// Если Регистратор может быть 20 видами документов — это 20 LEFT JOIN!
```

## Правило 13: Условие соединения vs условие ГДЕ в LEFT JOIN

**Почему:** В `ЛЕВОМ СОЕДИНЕНИИ` условие в `ГДЕ` и условие в `ПО` работают по-разному:
- Условие в `ПО` фильтрует **правую** таблицу ДО соединения — строки левой таблицы без пары остаются с NULL
- Условие в `ГДЕ` фильтрует **результат** ПОСЛЕ соединения — строки с NULL отбрасываются, превращая LEFT JOIN в INNER JOIN

### Правильно

```bsl
// Фильтр правой таблицы — в условии ON
Запрос.Текст =
"ВЫБРАТЬ
|   Номенклатура.Наименование,
|   ЕСТЬNULL(Цены.Цена, 0) КАК Цена
|ИЗ
|   Справочник.Номенклатура КАК Номенклатура
|       ЛЕВОЕ СОЕДИНЕНИЕ РегистрСведений.ЦеныНоменклатуры.СрезПоследних(&ДатаЦен,
|           ВидЦены = &ВидЦены) КАК Цены
|       ПО Номенклатура.Ссылка = Цены.Номенклатура";
// Все товары будут в результате, даже без цены
```

### Неправильно

```bsl
// ПЛОХО: фильтр по правой таблице в WHERE — LEFT JOIN фактически стал INNER JOIN
Запрос.Текст =
"ВЫБРАТЬ
|   Номенклатура.Наименование,
|   Цены.Цена
|ИЗ
|   Справочник.Номенклатура КАК Номенклатура
|       ЛЕВОЕ СОЕДИНЕНИЕ РегистрСведений.ЦеныНоменклатуры.СрезПоследних КАК Цены
|       ПО Номенклатура.Ссылка = Цены.Номенклатура
|ГДЕ
|   Цены.ВидЦены = &ВидЦены";  // Строки без цены (NULL) — отфильтруются!
```

## КОНТРОЛЬНЫЙ СПИСОК: Проверь перед коммитом

Используй этот список для проверки запросов:

### Структура запроса

- [ ] Сложные запросы разбиты на временные таблицы
