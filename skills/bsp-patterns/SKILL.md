# /bsp-patterns — Паттерны БСП и типовые решения

Перед реализацией задач по функционалу конфигурации проверь, есть ли готовое решение
в БСП. Этот скилл оркестрирует обращение к `/use-bsp` и сводит решение к конкретным
шагам.

## Правило

**Сначала БСП, потом самописный код.**

## Шаги

1. Опиши задачу 1–2 абзацами:
   - какие данные/объекты затрагиваются,
   - какие ограничения по платформе,
   - требования к совместимости.
2. Запусти `/use-bsp` и зафиксируй найденные подходы.
3. Если БСП покрывает задачу:
   - используй профильный скилл/шаг для внедрения (`epf-bsp-init`, `epf-bsp-add-command` и т.д.),
   - не создавай дублирующие реализации.
4. Если типовое решение отсутствует:
   - документируй почему БСП не подходит,
   - переходи к проектированию собственной логики по `/brainstorm`/`1c-feature-dev`.

## Полезные источники

- `use-bsp` — фактическая библиотека типовых практик и шаблонов.
- `mcp__onec_universal__search` (поиск по API терминов в коде/документации MCP-уровня).
- `get_member`/`get_members` — уточнение возможностей конкретных типовых объектов.

## Результат

- Решение должно быть трассируемо:
  - `BSP` (какой модуль/функция),
  - или `custom` (аргументация, почему шаблона нет).
