# /1c-test-runner — Тестирование бизнес-логики 1С

Совместимая точка запуска тестов для сценариев из внешнего набора скиллов.

## Что делает

- Проверяет, что проект готов к запуску тестов (база, профиль конфигурации, publish/обновления сделаны);
- выбирает канал запуска:
  - **предпочтительно** `test-runner` backend (`ENABLED_BACKENDS=...test-runner`);
  - или ручной/альтернативный путь через обычный цикл `/openspec-apply` + валидаторы + ручной контроль.

## Рекомендуемый порядок

1. Уточни цель:
   - `run all` / `run module` / `run specific`;
2. Зафиксируй окружение:
   - какая база и какой backend (`test-runner`),
   - есть ли подготовленный профиль тестов;
3. Выполни прогон:
   - если backend доступен, запусти соответствующий MCP-инструмент;
   - если нет backend — не блокируй задачу, предложи альтернативу:
     - локальная проверка изменений через `/openspec-apply` + `/cf-validate`/`/meta-validate`/`/form-validate`;
     - при доступе к веб/UI прогнать сценарий через `/web-test`.

## Контроль результатов

- результат/ошибки фиксируются в артефактной заметке;
- при падениях — `open`/`failure` и next-step fix list;
- для каждого упавшего кейса укажи:
  - шаг воспроизведения,
  - ожидаемый результат,
  - фактический вывод.

## Правила

1. Не считать фичу завершённой до зелёного отчёта по тест-кейсам/модулю.
2. Перед повторным прогоном при стабильных падениях — очистка/переинициализация тестовых данных.
3. Для бизнес-логики без `test-runner` используй `1c-web-session`/`/web-test` только как дополнительную проверку UI.
