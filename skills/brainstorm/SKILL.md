# /brainstorm — Идея → Проектирование → План

Оркестратор полного цикла для задач, где нужно сначала согласовать дизайн, а уже потом
выполнять изменения.

## Принцип

- **Одна точка согласования**: после показа дизайна и плана.  
- **Минимум вопросов до плана**: уточняем только критичные неоднозначности.
- **Skills-first**: на исполнение опираемся на существующие скиллы репозитория.

## Режимы

- `express`: 1–2 объекта, низкий риск.
- `standard`: 3–7 объектов/встроенных шагов.
- `full`: сложные/архитектурные изменения с зависимостями и рисками.

## Процесс

### 1) Сбор требований

Перед стартом уточни:
- цель и критерии успеха;
- затронутые объекты 1С (справочник/документ/регистр/формы/отчёты/расширение);
- ограничения (поддерживаемые конфигурации, БСП, совместимость);
- ограничения по времени/рискам.

Используй:
- `get_metadata` для определения структуры затронутых объектов;
- `bsl_search` или `bsl_search_tool` для проверки существующих точек интеграции;
- `graph_search`/`graph_related` при значительном пересечении зависимостей.

### 2) Подготовка дизайн-блока

Покажи пользователю единое сообщение со структурой:

- Контекст задачи
- Предположения
- Решение/подход
- Ключевые объекты
- Риски и ограничения
- План по шагам

Для `standard`/`full` добавь:
- какие альтернативы рассматривались и почему отклонены;
- какие проверки обязательны перед завершением.

### 3) Согласование

Заверши блоком: «Выполняю?».  
Дальше жди подтверждения для `standard/full` перед выполнением.

### 4) Исполнение после подтверждения

Если подтверждение получено:
- используй профильные скиллы по шагам;
- для 1С-изменений соблюдай существующие правила валидации:
  - `validate_query` для запросов,
  - `execute_code` для критичных скриптов,
  - соответствующие `*-validate` в конце этапов.
- фиксируй прогресс в `openspec/changes/<feature>/tasks.md`, если задача сложная.

### 5) Финальный отчёт

Верни итог в формате:

- Что сделано
- Какие скиллы/шаги применены
- Что проверено
- Что осталось / риски
- Следующий шаг

## Правила

1. Не начинать имплементацию до согласования дизайна на `standard/full`.
2. Для каждой задачи должны быть проверяемые критерии приёмки.
3. Избегай «супер-обобщённых» решений: всё должно ссылаться на конкретные объекты и файлы.
4. Не добавляй изменения без необходимости и не выходи за рамки scope, если не согласовано.
