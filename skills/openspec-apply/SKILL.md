# /openspec-apply — Реализация согласованного изменений

Используется после `openspec-proposal` и (желательно) `/write-plan`.

## Принцип

- Выполнять строго по `tasks.md` без отклонений от scope.
- После каждой задачи — валидационный шаг.
- Для каждого значимого изменения обновлять состояние задания.

## Вход

`$ARGUMENTS` — `openspec/changes/<feature-id>/tasks.md` или `<feature-id>`.

## Процесс

1. Найди `tasks.md`:
   - если передан путь — проверка существования,
   - если только `feature-id` — чтение `openspec/changes/<feature-id>/tasks.md`.
2. Прочитай связанные:
   - `design.md` и `proposal.md` (если есть),
   - исходные файлы/объекты из плана.
3. Выполняй задачи последовательно, не пропуская зависимости:
   - метаданные/формы/модули через профильные `*-compile`/`*-edit`/`execute_code`;
   - документируй результат в `tasks.md` (`[ ]` → `[x]`).
4. Для каждой задачи с риском выполняй верификацию:
   - `validate_query` для запросов;
   - профильный `*-validate` для 1С-артефактов.
5. После финальной задачи:
   - убедись, что все критичные acceptance criteria закрыты;
   - собери список изменённых файлов и итоговых проверок.

## Результат

Выведи:
- какие задачи выполнены;
- какие проверки пройдены/не пройдены;
- какие артефакты изменены;
- что осталось (если есть блокеры).

## Запреты

- Не запускать новую плановую декомпозицию внутри `/openspec-apply` — для этого снова ` /write-plan`.
- Не включать незаказанные изменения и не выходить за `tasks.md`.
