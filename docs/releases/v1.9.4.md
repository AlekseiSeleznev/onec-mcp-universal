# onec-mcp-universal v1.9.4

`v1.9.4` исправляет реальный install bug, найденный на финальном destructive reinstall после релиза `v1.9.3`.

## Что исправлено

- `setup.sh` больше не зависает на host-side health-check, если `localhost` на машине резолвится в `::1`, а gateway слушает только IPv4.
- `verify-install.sh` и `verify-install.ps1` переведены на IPv4 loopback для локальных health-check'ов.
- В install docs и asset-tests добавлена фиксация этого контракта: локальная проверка здоровья использует `127.0.0.1`, а пользовательский MCP endpoint по-прежнему остаётся `http://localhost:8080/mcp`.

## Причина

На clean reinstall `setup.sh` запускал `curl -sf http://localhost:8080/health`, и один из вызовов зависал в `SYN-SENT` к `::1:8080`. Контейнерный gateway в этот момент уже был healthy и доступен по IPv4. Из-за этого установка выглядела зависшей, хотя стек фактически поднялся.

## Подтверждённый результат

- `./setup.sh --with-bsl-graph` на fresh clone завершается полностью, без зависания на health-check;
- `./verify-install.sh` проходит на fresh clone;
- security acceptance из `v1.9.3` сохраняется:
  - у gateway нет mount на `docker.sock`;
  - `NAPARNIK_API_KEY` отсутствует в env контейнера gateway;
  - основной процесс `onec-mcp-gw` работает как UID `10001`;
  - `ALLOW_DANGEROUS_WITH_APPROVAL=false` в toolkit по умолчанию;
  - REST rate limiting возвращает `429`, `/mcp` не throttled;
- полный `pytest` на fresh clone снова проходит с `100%` coverage.
