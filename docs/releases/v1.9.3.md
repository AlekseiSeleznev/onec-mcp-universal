# onec-mcp-universal v1.9.3

`v1.9.3` завершает цикл `v1.9.x` после live clean-reinstall проверки на свежем клоне из GitHub.

## Что исправлено

- Unit-тесты `gateway/tests/test_docker_manager.py` и `gateway/tests/test_server.py` больше не зависят от случайного наличия живого `docker-control` рядом с тестовым процессом.
- Для direct/mock unit-тестов добавлена принудительная изоляция от sidecar; отдельные тесты HTTP-пути docker-control помечены marker `allow_docker_control`.
- В `gateway/pytest.ini` зарегистрирован marker `allow_docker_control`, чтобы test suite оставался чистым без warning'ов.
- GitHub Actions переведены на Node 24 path для JavaScript actions: `actions/setup-python@v6` и workflow-level `FORCE_JAVASCRIPT_ACTIONS_TO_NODE24=true`.

## Почему понадобился этот патч

После полного удаления локальной установки, fresh clone из GitHub и повторного `./setup.sh --with-bsl-graph` runtime-проверки прошли, но часть unit-тестов внезапно начала ходить в реальный `docker-control`, потому что live sidecar уже был поднят. В CI это не проявлялось, так как там sidecar вне тестового процесса не существовал.

`v1.9.3` делает test suite детерминированным и на чистом клоне, и в CI, и на машине разработчика с уже поднятым стеком.

## Подтверждённый результат

- fresh clone из GitHub: `./setup.sh --with-bsl-graph` проходит без ручных действий;
- `./verify-install.sh` даёт `PASS=11, WARN=0, FAIL=0`;
- `codex mcp list` содержит `onec-universal`;
- gateway запущен без `docker.sock`, `NAPARNIK_API_KEY` отсутствует в env контейнера;
- основной процесс `onec-mcp-gw` работает как UID `10001`;
- `onec-mcp-toolkit` получает `ALLOW_DANGEROUS_WITH_APPROVAL=false`;
- REST rate limiting возвращает `429`, `/mcp` не throttled;
- полный `pytest` на fresh clone снова проходит с `100%` coverage.
