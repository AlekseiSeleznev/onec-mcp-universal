# onec-mcp-universal v1.9.0

## Что изменилось

### Безопасность и runtime-политика

- Gateway больше не получает прямой доступ к `docker.sock`.
- Добавлен внутренний sidecar `docker-control`, через который gateway управляет runtime-контейнерами, читает Docker diagnostics и проксирует LSP stdio.
- Gateway запускается как non-root процесс внутри контейнера.
- `NAPARNIK_API_KEY` больше не прокидывается в `environment` gateway-контейнера и читается только из примонтированного `.env`.
- Dangerous write/exec path в toolkit теперь выключен по умолчанию. Для явного включения используйте `TOOLKIT_ALLOW_DANGEROUS_WITH_APPROVAL=true`.
- Добавлен rate limiting для REST API и dashboard-маршрутов:
  - `GATEWAY_RATE_LIMIT_ENABLED=true`
  - `GATEWAY_RATE_LIMIT_READ_RPM=120`
  - `GATEWAY_RATE_LIMIT_MUTATING_RPM=30`
- `MCP` endpoint `/mcp` и healthcheck `/health` намеренно не ограничиваются rate limiter'ом.

### Установка и compose

- `setup.sh` автоматически настраивает `DOCKER_CONTROL_URL`:
  - Linux: `http://localhost:8091`
  - Windows/macOS bridge path: `http://host.docker.internal:8091`
- `setup.sh` сохраняет устойчивость к sparse `.env` и корректно дописывает отсутствующие ключи.
- `verify-install.sh` и `verify-install.ps1` теперь отдельно проверяют доступность `docker-control`.
- `docker-compose.yml` и `docker-compose.windows.yml` синхронизированы с новым sidecar-сервисом.

### Документация и публикация

- README обновлён под новую safe-by-default модель и нейтрализован в верхнем описании.
- `CODEX.md` и dashboard docs (`ru/en`) обновлены под `docker-control`, rate limiting и dangerous opt-in.
- `Docker Build & Publish` публикует три образа:
  - `ghcr.io/alekseiseleznev/onec-mcp-universal`
  - `ghcr.io/alekseiseleznev/onec-mcp-universal-bsl-graph-lite`
  - `ghcr.io/alekseiseleznev/onec-mcp-universal-docker-control`

### Тесты

- Добавлены unit/asset-тесты для:
  - `DockerControlLspBackend`
  - rate limiting
  - HTTP/fallback веток `docker_manager`
  - новых workspace/path-веток `bsl_search`
  - explicit `lsp_backend_factory` ветки в `db_lifecycle`
  - exception-ветки `server._get_optional_services_status`
- Локальный CI-эквивалентный прогон:
  - `973 passed`
  - `line+branch coverage = 100.00%`
