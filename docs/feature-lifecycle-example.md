# Example: Полный lifecycle фичи через workflow-скиллы

Демонстрация, как выполнять фичу от идеи до закрытия без потери контекста и с
трассируемыми артефактами.

## Цель примера

Добавить новый реквизит `КонтактныйТелефон` в документ `ЗаказПродаж` и обработать
его в печатной форме `ЗаказПокупателя`.

## Шаг 0. Подготовка и базовое согласование

1. Вводим идею:

```text
/brainstorm "Добавить обязательное поле КонтактныйТелефон в документ ЗаказПродаж и вывести его в печатную форму"
```

2. Скилл:
- собирает контекст задачи;
- формирует `Goal / Scope / Design / Risks / Acceptance`;
- для `standard/full` просит согласование перед выполнением.

## Шаг 1. Формирование официального пакета изменений (OpenSpec)

```text
/openspec-proposal "Добавить КонтактныйТелефон в документ ЗаказПродаж"
```

3. Результат:
- создаёт `openspec/changes/<feature-id>/`
- пишет `proposal.md`, `design.md`, `tasks.md` (черновик)
- при наличии исходных ограничений добавляет источники данных в дизайн

## Шаг 2. Атомаризация плана

```text
/write-plan <feature-id>
```

4. Результат:
- обновляет/формирует `openspec/changes/<feature-id>/tasks.md`
- разбивает работу на короткие шаги с зависимостями и критериями `Done`
- для каждого шага указывает нужный профильный скилл:
  - `get_metadata`
  - `meta-compile`/`form-edit`/`meta-edit`
  - `mxl-info`/`mxl-compile`/`template-add`
  - соответствующие validate-скиллы

## Шаг 3. Реализация по плану

```text
/openspec-apply <feature-id>
```

5. Внутри выполнения для каждого шага:
- выполняет задачу строго по порядку;
- после шага запускает валидацию и отмечает `tasks.md` как `[x]`;
- на критичных участках применяет:
  - `validate_query` для проверок запросов;
  - `/cf-validate` / `/meta-validate` / `/form-validate` / `/mxl-validate` по затрагиваемым артефактам.

6. Финальный результат:
- `openspec/changes/<feature-id>/tasks.md` со статусами;
- список изменённых файлов;
- сводка по проверкам.

## Шаг 4. Архивация

```text
/openspec-archive <feature-id>
```

7. Условия:
- все задачи закрыты как выполненные;
- проверки пройдены;
- статус фиксируется в `openspec/changes/archive/<feature-id>/`.

## Ожидаемый выход

- Есть единый трек от идеи до закрытия;
- изменения не смешаны с не связанными задачами;
- `design.md`, `tasks.md`, `proposal.md` и отчётные метки служат источником правды по фиче;
- `session-*` помогают при долгих сеансах без потери контекста.
