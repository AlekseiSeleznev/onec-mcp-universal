# Пример полного жизненного цикла фичи через workflow-скиллы

Дата: 2026-04-21  
Статус: актуально для `v1.0.0`

Этот документ показывает, как вести фичу от идеи до закрытия без потери контекста и с трассируемыми артефактами.

## Цель примера

Добавить новый реквизит `КонтактныйТелефон` в документ `ЗаказПродаж` и использовать его в печатной форме `ЗаказПокупателя`.

## Шаг 0. Подготовка и базовое согласование

Стартуем с формулировки задачи:

```text
/brainstorm "Добавить обязательное поле КонтактныйТелефон в документ ЗаказПродаж и вывести его в печатную форму"
```

Что делает `brainstorm`:

- собирает контекст задачи;
- формирует рабочую рамку: цель, границы, дизайн, риски, критерии приёмки;
- для сложного сценария подводит к согласованию дизайна до начала реализации.

На этом шаге важно не писать код, а зафиксировать, что именно меняется и почему.

## Шаг 1. Формирование официального пакета изменений

Дальше оформляем изменение как отдельный формализованный пакет изменений:

```text
/openspec-proposal "Добавить КонтактныйТелефон в документ ЗаказПродаж"
```

Ожидаемый результат:

- создаётся `openspec/changes/<feature-id>/`
- появляются `proposal.md`, `design.md`, `tasks.md`
- в дизайн попадают исходные ограничения и предпосылки

Смысл этого шага — отделить фичу от остального шума и получить единый контейнер для её жизненного цикла.

## Шаг 2. Атомаризация плана

После оформления предложения нужно разложить работу на выполнимые шаги:

```text
/write-plan <feature-id>
```

Что должен дать результат:

- `tasks.md` с короткими атомарными шагами;
- зависимости между шагами;
- критерии завершения для каждого шага;
- указание, какими skills и инструментами выполнять каждый участок.

Для такого примера там обычно появятся ссылки на:

- `get_metadata`
- `meta-edit`, `form-edit`, `meta-compile`
- `mxl-info`, `mxl-compile`, `template-add`
- соответствующие validate-skills

## Шаг 3. Реализация по плану

Теперь можно переходить к исполнению:

```text
/openspec-apply <feature-id>
```

Правильное поведение здесь такое:

- задачи выполняются по порядку;
- после каждого шага запускается релевантная проверка;
- в `tasks.md` фиксируется прогресс;
- критичные изменения не остаются без validation.

Типичные проверки в этом примере:

- `validate_query` — если появляются или меняются запросы;
- `/cf-validate`, `/meta-validate`, `/form-validate`, `/mxl-validate` — по затронутым артефактам.

Ожидаемый промежуточный результат:

- `tasks.md` со статусами;
- список изменённых файлов;
- краткая сводка по проверкам.

## Шаг 4. Сохранение состояния по ходу работы

Если задача длинная или работу нужно прервать, включается путь сохранения состояния:

```text
/session-save
```

Это даёт:

- актуальный `session-notes.md`;
- конкретный `Next Action`;
- список уже сделанного и оставшегося.

При возвращении в задачу используется:

```text
/session-restore
```

А после завершения этапа полезно зафиксировать короткий итог:

```text
/session-retro
```

## Шаг 5. Архивация изменения

Когда все задачи закрыты и проверки пройдены, изменение архивируется:

```text
/openspec-archive <feature-id>
```

Условия для этого шага:

- все задачи завершены;
- проверки пройдены;
- состояние изменения можно считать закрытым;
- архивный след остаётся в `openspec/changes/archive/<feature-id>/`.

## Что должно получиться в хорошем сценарии

На выходе мы получаем:

- единый трек от идеи до закрытия;
- отсутствие смешивания фичи с не связанными задачами;
- `proposal.md`, `design.md`, `tasks.md` как рабочие артефакты, а не одноразовый текст;
- сохранение и восстановление состояния через `session-*`, если работа прерывалась;
- понятный журнал изменений: что задумали, как реализовали, чем проверили и когда закрыли.

## Почему этот пример важен

Этот сценарий показывает главную идею workflow-слоя:

- не прыгать сразу к произвольному набору низкоуровневых действий;
- сначала оформить изменение;
- потом разложить его на атомарные шаги;
- потом провести исполнение и проверки;
- затем закрыть и заархивировать жизненный цикл изменения.

Именно так workflow-слой превращает набор отдельных skills в управляемый процесс разработки.
