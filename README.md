# onec-mcp-universal

**Универсальный MCP-сервер для 1С:Предприятие**

Единый Docker-контейнер, реализующий протокол [Model Context Protocol (MCP)](https://modelcontextprotocol.io) для работы с платформой 1С:Предприятие. Обеспечивает AI-ассистентам (Claude Code, Cursor и др.) доступ к инструментам работы с живыми данными базы, навигации по BSL-коду и документации платформы через одно HTTP-соединение.

## Назначение

**MCP (Model Context Protocol)** — открытый протокол, позволяющий AI-ассистентам вызывать внешние инструменты. Вместо ответов на основе общих знаний ассистент получает возможность напрямую обращаться к базе 1С, анализировать код и документацию платформы.

Для работы с 1С существует несколько независимых MCP-серверов с разными транспортами и протоколами подключения. Данный проект агрегирует их в единую точку доступа: один URL, один контейнер, одна строка в настройках.

**Пример сценария использования:**

Запрос: *«Найди все места, где вызывается РассчитатьНДФЛ, проверь наличие ошибок и покажи документацию по используемым методам платформы»*

Система автоматически:
1. Находит все вызовы через BSL Language Server
2. Получает текст процедур и запускает диагностику
3. Возвращает описание методов из документации платформы

## Архитектура

Внутри контейнера запущено три специализированных сервера и Python-шлюз, агрегирующий их инструменты:

```
Claude Code / Cursor
      │
      │  HTTP :8080/mcp
      ▼
┌──────────────────────────────────────────────────────┐
│               onec-mcp-gateway (Python)              │
│                                                      │
│  ┌─────────────────┐   ┌────────────────────────┐   │
│  │  onec-toolkit   │   │   platform-context     │   │
│  │  :6003          │   │   :8081                │   │
│  │  Python/FastAPI │   │   Java/Spring Boot     │   │
│  │  8 инструментов │   │   5 инструментов       │   │
│  └─────────────────┘   └────────────────────────┘   │
│                                                      │
│  ┌──────────────────────────────────────────────┐   │
│  │  BSL Language Server + mcp-lsp-bridge        │   │
│  │  Java 21 + Go                                │   │
│  │  14 инструментов                             │   │
│  └──────────────────────────────────────────────┘   │
└──────────────────────────────────────────────────────┘
      │
      │  HTTP :6003  (обработка MCPToolkit → gateway)
      ▼
1С:Предприятие (локально или удалённый сервер)
```

Шлюз маршрутизирует каждый вызов инструмента к соответствующему бэкенду по точному совпадению имени. При недоступности одного из бэкендов остальные продолжают работу.

**Транспорты:**

| Бэкенд | Транспорт | Протокол |
|--------|-----------|---------|
| onec-toolkit | Streamable HTTP | `/mcp` |
| platform-context | SSE | `/sse` |
| bsl-lsp-bridge | stdio (subprocess) | MCP stdio |

## Инструменты

### Работа с живыми данными 1С (8 инструментов)

Требуется загруженная обработка **MCPToolkit** в базе 1С (см. [раздел ниже](#обработка-mcptoolkit)).

| Инструмент | Описание |
|-----------|---------|
| `execute_1c_query` | Выполнение произвольного запроса к базе |
| `execute_1c_code` | Выполнение произвольного кода на языке 1С |
| `get_metadata` | Структура метаданных конфигурации |
| `get_event_log` | Чтение журнала регистрации с фильтрацией |
| `get_object_by_link` | Получение данных объекта по ссылке |
| `get_link_of_object` | Получение ссылки на объект |
| `find_references_to_object` | Поиск ссылок на объект в базе |
| `get_access_rights` | Права доступа пользователей |

### Документация платформы 1С (5 инструментов)

Работают без подключённой базы. Используют локально установленную платформу 1С.

| Инструмент | Описание |
|-----------|---------|
| `search` | Полнотекстовый поиск по API платформы |
| `getMembers` | Методы и свойства указанного типа |
| `getMember` | Описание конкретного метода или свойства |
| `getConstructors` | Конструкторы типа |
| `info` | Информация о версии платформы |

### Навигация по BSL-коду (14 инструментов)

Требуется папка с выгруженными `.bsl`-файлами конфигурации.

| Инструмент | Описание |
|-----------|---------|
| `symbol_explore` | Поиск процедур, функций и переменных по имени |
| `definition` | Переход к определению символа |
| `hover` | Документация и тип символа |
| `call_hierarchy` | Дерево вызовов функции |
| `call_graph` | Граф вызовов между модулями |
| `document_diagnostics` | Диагностика файла: ошибки и предупреждения |
| `code_actions` | Доступные действия рефакторинга |
| `rename` | Безопасное переименование символа в проекте |
| `get_range_content` | Содержимое диапазона строк файла |
| `lsp_status` | Статус BSL Language Server |
| + 4 | Дополнительные LSP-операции |

### Системные инструменты (2 инструмента)

| Инструмент | Описание |
|-----------|---------|
| `get_server_status` | Состояние всех бэкендов и число доступных инструментов |
| `export_bsl_sources` | Выгрузка BSL-исходников конфигурации через `ibcmd` с последующей переиндексацией |

## Требования

### Обязательно

- **Docker** — версия 20.10 и выше
- **Три локально доступных Docker-образа** (скачиваются один раз):
  ```bash
  docker pull roctup/1c-mcp-toolkit-proxy:latest
  docker pull ghcr.io/alkoleft/mcp-bsl-platform-context:latest
  # mcp-lsp-bridge-bsl:latest — собирается из alkoleft/mcp-bsl-lsp-bridge
  ```
- **1С:Предприятие** — платформа установлена на машине, где запускается контейнер (для `platform-context` и `ibcmd`)

### Для полной функциональности

- **Папка с BSL-файлами** конфигурации (`.bsl`) — для инструментов навигации по коду. Выгружается автоматически через инструмент `export_bsl_sources` или кнопку в обработке MCPToolkit.
- **Обработка MCPToolkit** в базе 1С — для инструментов работы с живыми данными.
- **Пользователь с правами администратора** в базе 1С — для выгрузки BSL-исходников через `ibcmd`.

### Поддерживаемые режимы подключения к 1С

| Режим | Поддержка |
|-------|---------|
| Тонкий клиент (`.exe`) через любой URL | ✅ |
| Файловая база | ✅ |
| Клиент-серверная база | ✅ |
| Веб-клиент (браузер) | ❌ Native AddIn не поддерживается |

## Развёртывание

### Шаг 1. Клонирование репозитория

```bash
git clone https://github.com/AlekseiSeleznev/onec-mcp-universal.git
cd onec-mcp-universal
```

### Шаг 2. Сборка образа

Сборка выполняется полностью из локально доступных образов. Интернет требуется только при первоначальном скачивании образов.

```bash
docker build -f all-in-one/Dockerfile -t onec-mcp-universal .
```

### Шаг 3. Запуск контейнера

```bash
docker run -d \
  --name onec-mcp \
  --restart unless-stopped \
  -p 8080:8080 \
  -p 6003:6003 \
  -v /opt/1cv8:/opt/1cv8:ro \
  -v /путь/к/bsl:/projects:rw \
  -e PLATFORM_PATH=/opt/1cv8/x86_64/8.3.27.2074 \
  -e BSL_WORKSPACE=/projects \
  onec-mcp-universal:latest
```

Замените:
- `/путь/к/bsl` — путь к папке с `.bsl`-файлами конфигурации (или пустая папка, если планируется выгрузка через `export_bsl_sources`)
- `/opt/1cv8/x86_64/8.3.27.2074` — путь к вашей версии платформы 1С

### Шаг 4. Проверка готовности

Контейнеру требуется ~30–60 секунд для запуска BSL Language Server.

```bash
docker logs onec-mcp --tail 20
```

Признак успешного запуска — строка `Backends ready: 3/3`.

## Подключение к Claude Code

Добавьте в `~/.claude/settings.json` в раздел `mcpServers`:

```json
{
  "mcpServers": {
    "onec": {
      "type": "http",
      "url": "http://localhost:8080/mcp"
    }
  }
}
```

Перезапустите Claude Code. Для проверки выполните запрос:

> *Вызови инструмент get_server_status*

Ожидаемый ответ:
```json
{
  "bsl-lsp-bridge":   {"ok": true, "tools": 14},
  "onec-toolkit":     {"ok": true, "tools": 8},
  "platform-context": {"ok": true, "tools": 5}
}
```

## Конфигурация

Параметры задаются через переменные окружения при запуске контейнера.

| Переменная | Значение по умолчанию | Описание |
|-----------|----------------------|---------|
| `PLATFORM_PATH` | `/opt/1cv8/x86_64/8.3.27.2074` | Путь к версии платформы 1С |
| `BSL_WORKSPACE` | `/projects` | Путь к BSL-файлам внутри контейнера |
| `BSL_JAVA_XMX` | `6g` | Максимальный объём памяти BSL Language Server |
| `BSL_JAVA_XMS` | `2g` | Начальный объём памяти BSL Language Server |
| `IBCMD_PATH` | `/opt/1cv8/x86_64/8.3.27.2074/ibcmd` | Путь к утилите `ibcmd` |
| `ONEC_TIMEOUT` | `180` | Таймаут выполнения кода/запросов 1С (секунды) |
| `LOG_LEVEL` | `INFO` | Уровень логирования: `DEBUG`, `INFO`, `WARNING`, `ERROR` |
| `ENABLED_BACKENDS` | `onec-toolkit,platform-context,bsl-lsp-bridge` | Активные бэкенды |

**Требования к памяти:**

| Конфигурация | RAM |
|-------------|-----|
| Только platform-context | ~500 МБ |
| + onec-toolkit | ~700 МБ |
| + BSL LS (небольшая конфигурация) | ~3 ГБ |
| + BSL LS (ZUP/ERP, 7000+ файлов) | ~8 ГБ |

**Запуск без BSL-навигации** (если BSL-файлы недоступны):

```bash
-e ENABLED_BACKENDS=onec-toolkit,platform-context
```

## Выгрузка BSL-исходников

### Через AI-ассистента (инструмент `export_bsl_sources`)

Задайте запрос в Claude Code:

> *Выгрузи BSL-исходники базы ZUP. Строка подключения: Srvr=server1c;Ref=zup;Usr=admin;Pwd=pass*

Или вызовите инструмент напрямую:

```
export_bsl_sources(
  connection="Srvr=server1c;Ref=zup;Usr=admin;Pwd=pass",
  output_dir="/projects"
)
```

### Через обработку MCPToolkit (вкладка «Выгрузка BSL»)

Обработка MCPToolkit содержит вкладку **«Выгрузка BSL»** с кнопками:
- **Определить автоматически** — заполняет поля сервера и базы из текущего подключения
- **Выгрузить BSL-исходники** — отправляет запрос на выгрузку в контейнер

Подходит для случаев, когда доступ к Конфигуратору отсутствует, но есть пользователь с правами администратора.

### Форматы строки подключения

| Тип базы | Строка подключения |
|---------|-------------------|
| Клиент-серверная | `Srvr=server1c;Ref=имя_базы;Usr=пользователь;Pwd=пароль` |
| Файловая | `File=/путь/к/базе` |

### Процесс после выгрузки

1. `ibcmd` экспортирует объекты конфигурации в BSL-формат в `/projects`
2. Gateway уведомляет BSL Language Server об изменении файлов
3. BSL Language Server переиндексирует файлы (~1–5 минут для крупных конфигураций)
4. Инструменты навигации становятся доступны

### Необходимые права

Пользователь 1С должен иметь роль **«Администратор»** или право **«Конфигурирование данных»**. Доступ к Конфигуратору не требуется.

## Работа с несколькими базами

### Инструменты данных

Инструменты `execute_1c_query`, `get_metadata` и другие работают с любой базой. Обработка MCPToolkit может быть одновременно открыта в нескольких базах — каждая подключается к одному контейнеру независимо.

### BSL-навигация

BSL Language Server работает с одной папкой файлов одновременно.

**Переключение на другую базу:**

1. Выгрузите исходники новой базы через `export_bsl_sources` или вкладку в MCPToolkit
2. Файлы в `/projects` будут перезаписаны
3. BSL Language Server переиндексирует новые файлы (~1–5 минут)

**Параллельная работа с двумя базами:**

Запустите два контейнера на разных портах:

```bash
# Контейнер для первой базы
docker run -d --name onec-mcp-zup \
  -p 8080:8080 -p 6003:6003 \
  -v /opt/1cv8:/opt/1cv8:ro \
  -v /bsl/zup:/projects:rw \
  -e PLATFORM_PATH=/opt/1cv8/x86_64/8.3.27.2074 \
  onec-mcp-universal:latest

# Контейнер для второй базы
docker run -d --name onec-mcp-erp \
  -p 8090:8080 -p 6013:6003 \
  -v /opt/1cv8:/opt/1cv8:ro \
  -v /bsl/erp:/projects:rw \
  -e PLATFORM_PATH=/opt/1cv8/x86_64/8.3.27.2074 \
  onec-mcp-universal:latest
```

```json
{
  "mcpServers": {
    "onec-zup": { "type": "http", "url": "http://localhost:8080/mcp" },
    "onec-erp": { "type": "http", "url": "http://localhost:8090/mcp" }
  }
}
```

> В версии 0.2 запланировано встроенное переключение профилей через инструмент `switch_profile`.

## Обработка MCPToolkit

Инструменты работы с живыми данными используют внешнюю обработку **MCPToolkit**, загружаемую непосредственно в базу 1С.

### Схема взаимодействия

```
Claude Code        Контейнер (:6003)       1С (тонкий клиент)
     │                    │                        │
     │── вызов ──────────►│                        │
     │                    │── команда ────────────►│
     │                    │   (ожидание)           │ (выполнение)
     │                    │◄── результат ──────────│
     │◄── ответ ──────────│                        │
```

Обработка периодически опрашивает `http://адрес-контейнера:6003/1c/poll`, получает команды, выполняет их и возвращает результаты. Код клиентской части (`&НаКлиенте`) выполняется локально — HTTP-запросы адресованы к локальной машине.

### Установка

1. Скачайте файл `MCPToolkit.epf` из папки [`1c/`](./1c/) данного репозитория
2. Откройте в 1С через **Файл → Открыть**
3. Укажите адрес сервера: `http://localhost:6003`
4. Нажмите **Подключиться**

## Структура репозитория

```
onec-mcp-universal/
│
├── all-in-one/
│   ├── Dockerfile           # Multi-stage сборка из трёх локальных образов
│   ├── entrypoint.py        # Python process manager (запуск всех процессов)
│   └── supervisord.conf     # Конфигурация для режима с supervisord
│
├── gateway/
│   ├── Dockerfile           # Образ шлюза для режима docker-compose
│   ├── requirements.txt
│   └── gateway/
│       ├── __main__.py      # Точка входа (uvicorn)
│       ├── server.py        # Starlette-приложение, маршруты, lifespan
│       ├── mcp_server.py    # MCP Server: список инструментов, маршрутизация
│       ├── config.py        # Настройки из переменных окружения
│       └── backends/
│           ├── base.py          # Абстрактный класс бэкенда
│           ├── http_backend.py  # Streamable HTTP и SSE клиенты
│           ├── stdio_backend.py # stdio (subprocess или docker exec)
│           └── manager.py       # Менеджер бэкендов, маршрутизация по имени
│
├── docker-compose.yml       # Режим раздельных контейнеров
├── .env.example             # Шаблон конфигурации
└── start.sh                 # Запуск в host-режиме
```
